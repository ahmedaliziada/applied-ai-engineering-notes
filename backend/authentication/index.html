    Authentication & JWT

    Authentication and authorization are core concepts in backend engineering and production AI systems.

    Authentication vs Authorization

    Authentication

    Authentication answers:

    Who are you?

    Examples:

    • Login
    • Email/password verification
    • Identity verification

    Authorization

    Authorization answers:

    What are you allowed to do?

    Examples:

    • Can access premium AI model?
    • Has enough quota?
    • Is admin?
    • Enterprise user?

    Authentication Flow

    Detailed View

    flowchart TD

A[Step 1 - Login]
--> B[Step 2 - Server Verifies]

B --> C[Check Email and Password]

C --> D[Authenticated User]

D --> E[Step 3 - Server Creates Token]

E --> F[Step 4 - Client Stores Token]

F --> G[Step 5 - Every Request Includes Token]

G --> H[Backend Verifies Token]

H --> I[Allow or Deny Request]

    Helicopter View

    flowchart LR

A[Browser / Frontend]
--> B[Send Email + Password Once]

B --> C[Backend]

C --> D[JWT Token]

D --> A

    Why Do We Need Tokens?

    HTTP is stateless.

    Each request is independent.

    The server does not automatically remember previous requests.

    Tokens help applications identify users across multiple requests.

    Types of Tokens

    1. Random Tokens

    Example:

    x7A91P0Q

    The server stores token-related information inside a database or session store.

    2. JWT (JSON Web Token)

    JWT stores information directly inside the token payload.

    Example:

    xxxxx.yyyyy.zzzzz

    Payload example:

    {
  "user_id": 7,
  "name": "Ahmed",
  "role": "admin"
}

    JWT Structure

    JWT contains 3 parts:

    HEADER.PAYLOAD.SIGNATURE

    Header

    Contains token metadata.

    Example:

    {
  "alg": "HS256",
  "typ": "JWT"
}

    Payload

    Contains user-related information.

    Example:

    {
  "user_id": 7,
  "role": "admin",
  "exp": 1712345678
}

    Signature

    The server signs the token using a secret key.

    Conceptually:

    SIGN(header + payload, secret_key)

    This prevents unauthorized modifications.

    Why JWT Is Powerful

    The server does not need to store authentication session state.

    The token itself contains the information.

    This enables scalable authentication systems.

    JWT Security

    JWT is usually signed, not encrypted.

    Anyone can read the payload, but they should not be able to modify it.

    Never store:

    • passwords
    • API secrets
    • highly sensitive data

    inside JWT payloads.

    Sessions vs JWT

    flowchart LR

A[Client]
--> B[Server]

B --> C[(Session Store)]

C --> B

    In session-based authentication:

    • the server stores session state
    • the backend remembers users
    • scaling becomes harder

    JWT Stateless Authentication

    flowchart LR

A[Client + JWT]
--> B[Server A]

A --> C[Server B]

A --> D[Server C]

    In JWT authentication:

    • any server can validate the token
    • authentication becomes easier to scale
    • no centralized session store is required

    What Is State?

    State = information that must persist between requests.

    Example:

    Ahmed is logged in

    A traditional session-based system stores this information on the server.

    This is called a stateful system.

    The Scaling Problem

    In cloud and distributed systems:

    • multiple servers exist
    • requests may hit different servers
    • sessions become harder to synchronize

    Example:

    Load Balancer
    ↓
Server A
Server B
Server C

    JWT and Stateless Authentication

    JWT enables stateless authentication.

    Any server can validate the token independently.

    This makes scaling easier in:

    • cloud systems
    • microservices
    • AI APIs
    • serverless environments

    Important Note

    Stateless does NOT mean:

    No database exists.

    Databases are still used for:

    • users
    • quotas
    • billing
    • refresh tokens
    • revoked tokens

    However:

    Authentication session state itself does not need to be stored.

    Request Example

    GET /profile
Authorization: Bearer JWT_TOKEN

    The backend verifies the token before processing the request.

    Common HTTP Authentication Status Codes

    Status Code Meaning
    200 Success
    401 Unauthorized
    403 Forbidden

    Access Token vs Refresh Token

    Access Token

    Short-lived token used for API requests.

    Example:

    • 15 minutes
    • 1 hour

    Refresh Token

    Used to generate new access tokens without logging in again.

    Why This Matters for AI Systems

    Modern AI systems rely heavily on:

    • APIs
    • authentication
    • permissions
    • quotas
    • scalable cloud infrastructure

    Example:

    POST /chat

    The backend needs to determine:

    • who the user is
    • whether they are allowed to access the AI system
    • quota limits
    • enterprise permissions

    JWT is commonly used to solve this problem in production systems.

    Key Takeaways

    • HTTP is stateless
    • Tokens help identify users
    • JWT stores data inside the token payload
    • JWT enables scalable authentication systems
    • Authentication and authorization are different concepts
    • Stateless authentication is important for cloud-native systems
    • AI systems rely heavily on authentication and API security
